核心指令：根据以下 Sprint Backlog 和架构设计，为每个 Story 制定测试策略，明确测试类型、优先级和覆盖范围。

### 一、Story 测试分级

根据 Story 的业务风险和技术复杂度，划分测试等级：

| 测试等级 | 适用条件 | 测试类型组合 |
|:--|:--|:--|
| L1 关键 | 涉及资金/支付/权限/核心业务流程 | 单元测试 + 集成测试 + E2E 测试 + 安全测试 + 性能测试 |
| L2 重要 | 涉及多模块交互/外部系统集成/状态流转 | 单元测试 + 集成测试 + E2E 测试 |
| L3 一般 | 单模块 CRUD/配置管理/简单查询 | 单元测试 + 集成测试 |
| L4 低风险 | 文案修改/样式调整/非功能性优化 | 单元测试 + 冒烟测试 |

### 三、各测试类型的执行标准

#### 3.1 单元测试（Code Agent 负责，QA Agent 验证）

| 检查项 | 标准 |
|:--|:--|
| 覆盖率 | L1/L2 Story ≥ 85%，L3/L4 Story ≥ 70% |
| 分支覆盖 | 核心业务逻辑分支覆盖率 ≥ 80% |
| 边界值 | 所有数值型入参必须覆盖边界值（最小值、最大值、零、负数） |
| 异常路径 | 每个 public 方法至少覆盖 1 个异常路径 |
| Mock 规范 | 外部依赖必须 Mock，禁止单元测试依赖真实外部服务 |

#### 3.2 集成测试（QA Agent 负责）

| 检查项 | 标准 |
|:--|:--|
| 接口契约 | 每个 API 接口至少 1 个正向用例 + 1 个参数校验用例 + 1 个权限校验用例 |
| 数据库交互 | 验证 CRUD 操作的数据一致性，包括事务回滚场景 |
| 外部集成 | 使用 Mock/Stub 验证外部系统交互的请求格式和响应处理 |
| 状态流转 | 验证每个合法的状态转换路径，以及非法转换的拒绝 |

#### 3.3 E2E 测试（QA Agent 负责）

| 检查项 | 标准 |
|:--|:--|
| 场景覆盖 | 覆盖 BA 验收用例中的所有正常场景 + 关键异常场景 |
| 角色覆盖 | 每个涉及的角色至少执行 1 条完整业务路径 |
| 数据贯穿 | 从数据创建到最终状态的全链路验证 |
| 浏览器兼容 | L1 Story 需覆盖 Chrome + Firefox + Safari |

#### 3.4 性能测试（仅 L1 Story）

| 检查项 | 标准 |
|:--|:--|
| 响应时间 | 核心 API P95 响应时间 ≤ 500ms，P99 ≤ 1000ms |
| 吞吐量 | 满足预期并发用户数下的 TPS 要求 |
| 资源占用 | CPU ≤ 70%，内存 ≤ 80%（压测期间） |
| 稳定性 | 持续压测 30 分钟无内存泄漏、无错误率上升 |

### 四、测试通过/失败判定标准

| 测试等级 | 通过条件 | 允许的遗留问题 |
|:--|:--|:--|
| L1 | 所有用例通过，0 高危 + 0 中危缺陷 | 仅允许低优先级 UI 类问题 |
| L2 | 所有用例通过，0 高危缺陷 | 允许 ≤ 2 个中危缺陷（需记录并排入下个 Sprint） |
| L3 | 核心用例通过，0 高危缺陷 | 允许 ≤ 3 个中危缺陷 |
| L4 | 冒烟测试通过 | 允许非阻塞性缺陷 |
