## 数据传递契约

定义各角色之间交接物的数据格式规范，Flow Agent 在每次数据传递时进行 schema 校验。

---

### 一、契约校验原则

1. 上游角色输出前，Flow Agent 校验输出是否符合契约 schema
2. 校验不通过 → 阻断传递，要求上游修正
3. 校验通过 → 传递给下游，下游可信任数据格式

#### PO → Arch Copilot（Sprint 规划触发）

PO 只传 Sprint Goal 和选中的 Story ID，Arch Copilot 根据 Story ID 自行从统一存储读取 BA 产出物（需求文档、场景列表、功能列表、依赖分析、状态梳理等）。

```json
{
  "sprint_selection": {
    "sprint_id": "string, 格式 SP-{seq}",
    "sprint_goal": "string, 必填",
    "start_date": "string, ISO 8601",
    "end_date": "string, ISO 8601",
    "selected_story_ids": ["string, Story ID"],
    "priority_overrides": [
      {
        "story_id": "string",
        "priority": "enum: must|should|could",
        "reason": "string, 可选"
      }
    ],
    "notes": "string, 可选"
  }
}
```

> Arch Copilot 按 story_id 从统一存储读取的 BA 产出物包括：requirements_doc、scenarios、function_list、external_dependencies、status_list。这些数据的 schema 见上方 BA Copilot → SM Agent 契约。

### 四、校验失败处理

| 校验失败类型 | 处理方式 |
|:--|:--|
| 必填字段缺失 | 阻断传递，返回缺失字段列表，要求上游补充 |
| 字段类型不匹配 | 阻断传递，返回类型错误详情，要求上游修正 |
| 枚举值不在范围内 | 阻断传递，返回合法枚举值列表 |
| 数据逻辑矛盾 | 阻断传递，返回矛盾描述（如 passed=5 但 total=3） |
