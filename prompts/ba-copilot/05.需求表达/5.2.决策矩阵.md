## 决策矩阵生成提示词

### **核心任务**基于业务流程单据的状态列表

状态列表如下：

【$status_list】，

为其生成一套解耦合的、角色为中心的决策矩阵。

## 状态解耦合设计原则

### 1.
状态维度分离

为避免状态膨胀导致矩阵难以维护，请将业务状态按以下维度拆分：

- **流程维度状态**：表示单据在业务流程中的位置

- **数据维度状态**：表示单据数据的完整性或质量状态

- **资源维度状态**：表示相关资源（如资金、物料、人员）的状态

- **时间维度状态**：表示单据在时间轴上的状态（如超时、待处理）

### 2.
状态分层设计

采用"主状态-子状态"的分层结构：

```

主状态（核心流程）

├ ─子状态（流程细节）

├ ─扩展状态（附加属性）

└─标记状态（特殊标记）

```

### 3.
状态组合原则

在决策矩阵中，主要使用主状态组合。子状态和扩展状态通过条件编号进行引用，避免矩阵列数膨胀。

## 输出结构要求

### 第一部分：状态维度分析

分析输入的【$Status_List】，按以下维度重新组织状态：

1. **核心流程状态（主状态）** -定义3-6个核心状态，覆盖业务流程主线

-
示例：草稿、审核中、已批准、执行中、已完成、已关闭

2. **扩展状态（条件状态）** -将其他状态分类为扩展状态

-
通过条件编号在主矩阵中引用

-
分类示例：

-
数据状态：数据完整、数据待补充、数据异常

-
资源状态：资源待分配、资源已锁定、资源不足

-
时间状态：正常、即将超时、已超时

3. **状态映射关系** -建立主状态与扩展状态的映射关系

-
说明状态组合的业务含义

### 第二部分：按角色和状态维度分解的矩阵

#### 设计原则

1. **每个角色一个主矩阵**：基于核心流程状态

2. **条件引用扩展状态**：在单元格中使用编号引用扩展状态条件

3. **分离关注点**：不同业务关注点使用不同维度的状态

#### 矩阵类型

**A.核心流程权限矩阵（每个角色一个）**```
|操作/核心状态|草稿|审核中|已批准|执行中|已完成|已关闭|
|:--|:--|:----|:----|:----|:----|:----|
|操作1 | | | | | | |
|操作2 | | | | | | |
```

**B.数据状态权限矩阵（可选）**当数据状态对权限有重要影响时生成：

```
|操作/数据状态|数据完整|数据待补充|数据异常|
|:--|:--|:--|:--|
|操作1 | | | |
|操作2 | | | |
```

**C.资源状态权限矩阵（可选）**当资源状态对权限有重要影响时生成：

```
|操作/资源状态|资源待分配|资源已锁定|资源不足|
|:--|:--|:--|:--|
|操作1 | | | |
|操作2 | | | |
```

### 第三部分：角色决策矩阵集合

#### 1.
申请人/创建者权限矩阵

-
核心操作：创建、编辑、提交、取消、查看历史、重新打开

-
重点关注：创建到提交阶段的权限递减

#### 2.
审核人/审批人权限矩阵

-
核心操作：审核通过、审核驳回、转派、加签、查看详情

-
重点关注：审核阶段的操作权限与约束

#### 3.
执行人/操作员权限矩阵

-
核心操作：接收任务、开始执行、更新进度、完成执行、报告问题

-
重点关注：执行阶段的操作权限与数据记录

#### 4.
系统自动操作矩阵

-
核心操作：状态自动转换、超时处理、通知发送、数据校验

-
重点关注：自动化触发的条件和时机

#### 5.
系统管理员权限矩阵

-
核心操作：强制状态变更、重新分配、数据修正、流程配置

-
重点关注：应急处理和系统维护的超级权限

### 第四部分：状态条件与业务规则说明

#### A.
扩展状态条件说明

对所有矩阵中引用的扩展状态条件进行详细说明：

``` (1)条件说明：当单据处于"数据待补充"状态时，申请人可编辑

(2)条件说明：当资源状态为"资源已锁定"时，执行人可开始执行

```

#### B.
状态组合规则

说明不同维度状态的组合规则和业务含义：

```

规则1：核心状态为"执行中"且资源状态为"资源不足"时，系统自动发送预警

规则2：核心状态为"审核中"且时间状态为"已超时"时，自动转派给上级

```

#### C.
状态转换规则

1. **主状态转换规则**：驱动核心业务流程的转换

2. **扩展状态转换规则**：在特定条件下的状态变更

3. **状态同步规则**：不同维度状态之间的同步关系

### 第五部分：跨角色状态协作模型

#### 1.
状态权限传递图

可视化展示状态变更时权限在不同角色间的传递关系。

#### 2.
状态冲突解决规则

当不同角色对同一单据有冲突操作时的处理规则：

-
时间戳优先规则

-
角色优先级规则

-
状态锁定机制

#### 3.
状态异常处理流程

定义各种异常状态的处理流程和角色职责。

### 第六部分：完整状态视图与索引

#### A.
状态全量清单

列出所有状态及其分类：

``` 1.核心流程状态

-
草稿、审核中、已批准、执行中、已完成、已关闭

2.数据维度状态

-
数据完整、数据待补充、数据异常

3.资源维度状态

-
资源待分配、资源已锁定、资源执行中、资源已释放

4.时间维度状态

-
正常、即将超时、已超时

```

#### B.
状态组合索引

创建状态组合到矩阵的索引，方便查找：

``` "审核中+数据待补充" →申请人可编辑，审核人可驳回

"执行中+资源不足" →执行人可报告问题，系统发送预警

```

#### C.
状态查询映射表

提供从原始状态到新状态维度的映射关系。

## 示例：采购申请单状态解耦合

### 原始状态列表

草稿、已提交、部门审核中、财务审核中、已批准、预算检查中、供应商选择中、合同拟定中、已下单、发货中、部分收货、已收货、验收中、验收通过、付款申请中、已付款、已完成、已关闭、已驳回、供应商驳回、合同驳回、验收驳回、已取消

### 解耦合后状态结构

#### 核心流程状态

1.草稿→创建阶段

2.审核中→审批阶段（合并部门审核、财务审核）

3.已批准→准备阶段（合并预算检查、供应商选择、合同拟定）

4.执行中→执行阶段（合并已下单、发货中、部分收货、已收货）

5.验收中→验收阶段（合并验收中、验收通过）

6.已完成→完成阶段（合并付款申请、已付款、已完成、已关闭）

#### 扩展状态条件

- (1)
审批子状态：部门审核、财务审核

- (2)
执行子状态：订单处理、物流运输、货物接收

- (3)
异常状态：各种驳回、取消

- (4)
资源状态：预算充足、预算不足、供应商选定

- (5)
时间状态：正常、即将超时、已超时

#### 申请人核心权限矩阵

|操作/核心状态|草稿|审核中|已批准|执行中|验收中|已完成|
|:--|:--|:--|:--|:--|:--|:--|
|创建申请|✓| | | | | |
|编辑申请|✓| (1) | | | | |
|提交审核|✓| | | | | |
|取消申请|✓| (3) | (3) | | | |
|查看详情|✓|✓|✓|✓|✓|✓|
|补充材料| | (1) | (2) | | | |

**条件说明：** (1)仅当审批子状态为"部门审核"且审核人未处理时

(2)当处于"供应商选择中"子状态时

(3)仅当执行子状态不涉及已下单或已发货时

## 实施指导

### 1.
状态解耦合步骤

1.识别核心业务流程主线，提取3-6个关键状态

2.将其他状态归类为扩展状态或子状态

3.定义状态维度（流程、数据、资源、时间）

4.建立状态映射关系

### 2.
矩阵设计步骤

1.为每个角色设计核心流程权限矩阵

2.识别需要特殊处理的维度，创建补充矩阵

3.使用条件编号引用扩展状态

4.确保所有原始状态都被映射到新结构中

### 3.
验证标准

-
核心矩阵列数不超过8列

-
每个扩展状态都有明确的引用条件

-
所有原始业务规则都被保留

-
状态转换逻辑清晰且完整
