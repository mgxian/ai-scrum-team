### 业务流程识别与结构化分析指令

#### 核心任务
根据以下内容：

业务需求如下：

【$optimized_requirements】

角色列表如下：

【$role_list】

识别并结构化分析其中的业务流程，输出符合标准的流程PlantUML代码。

---

#### 关键原则（通用业务原则，适用于所有领域）

1. **数据对象命名原则**所有流程处理的数据模型（dataModel）名称，必须源自数据库表设计中的中文表名，严禁随意创造。

2. **流程独立性原则**- **阶段拆分**：涉及多个阶段的流程应分阶段拆分为独立流程（例如：问卷编辑审批与问卷投放收集必须分为两个流程图）。
- **操作类型拆分**：增、删、改、查、注销、启用/禁用等基础资源管理操作，必须拆分为独立的CRUD流程，严禁嵌入一般审批或简单流程中。
- **辅助功能剥离**：所有跨流程汇总、统计、财务分析、报表生成等非核心业务操作，应设计为独立的定时任务或后台作业，**不得**出现在任何业务流程图中。

3. **角色职责合并原则**流程中的角色泳道应根据实际业务职责定义，避免过度细化。**凡是在业务逻辑中可由同一人/岗位完成的连续职责，必须合并为同一角色**（例如：小企业用车流程中派车员与审批人实为同一角色，应合并为派车管理员）。严禁为展示操作顺序而人为拆分角色。

4. **外部数据不纳入原则**- 若流程所需的基础数据（如车辆信息、司机信息、物料清单等）来源于其他系统同步或静态配置，流程中**不应包含**对这些数据的申请、审批、修改等管理步骤棗假设这些数据已就绪。
- 若业务确实需要维护这些基础数据，则必须将其独立设计为CRUD流程，**不得**与核心业务流程图混用。

5. **环节独立性原则**如合同、支付、注册等通用且独立的业务环节，应优先分析为简单流程或子流程，避免在主流程中展开全部细节。

---

#### 流程类型定义与步骤规范

你必须严格遵循以下定义对流程进行分类和步骤设计：

| 流程类型 | 定义与用途 | 允许的步骤类型 (`stepType`) |
| :--- | :--- | :--- |
| 1. 一般审批流程 | 必须包含审批环节的多步骤流程。可分为`主流程`和`子流程`。 | 必须按顺序包含：`pre` (预先处理) -> `apply` (申请) -> `approve` (审批) -> `execute-start` (开始执行) -> `execute-end` (结束执行) -> `post` (后续处理)。不可缺失。 |
| 2. 简单流程 | 不涉及审批的多步骤业务场景。如：注册、发布、配置、数据收集等。 | 限定于：`info`, `process`, `agree`, `back-end`, `credential`, `command`, `receive`, `form`, `publish`。 |
| 3. CRUD流程 | 对基础资源（如车辆、司机、规则）进行增删改查管理的操作流程要拆分成四个流程，并且如果可以从其他系统获取的信息不需要整理流程。 | 固定为：`crud`。不得包含一般审批流程的步骤。 |
| 4. 子流程 | 用于处理主流程中独立、可分离的特定环节（如出差流程中的订机票、办签证）。不应将主流程的连续步骤拆分为多个子流程 | 根据其性质，遵循上述1或2的步骤规范。 |

#### 详细规则说明（PlantUML 规范）

请为识别出的每个业务流程，严格按照以下`PlantUML`格式输出代码块。确保语法正确，没有冗余，可在任何标准PlantUML编辑器（如PlantUML在线服务器、VS Code插件）中直接渲染。

##### 1. 泳道图基本结构
- 使用 `|角色名|` 语法为每个角色创建独立的泳道。
- 泳道顺序按照角色在流程中出现的重要性或逻辑顺序排列。
- 每个动作节点必须放置在正确的角色泳道中。

##### 2. 节点生成规则
- **开始与结束**：
- 流程图必须以 `start` 节点开始。
- 所有流程路径必须以 `stop` 结束。
- 结束节点不归属于任何特定泳道。
- **动作节点**：
- 每个动作节点使用 `:动作名称;` 格式。
- 动作名称简短规范（2~4个中文简体字），如：提交申请、审批、执行任务。
- 动作节点必须放置在对应角色的泳道内。
- **判断节点**：
- 判断条件必须是简洁的是否...表达式，如：是否通过？、是否合理？
- 判断节点放置在做出判断的角色泳道中。
- 格式：`if (是否条件？) then (是)` 和 `else (否)`
- 判断后直接连接下一个节点，不在分支线上写动作。
- 判断节点（菱形）内不能有空值的情况。

##### 3. 流程结构规则
- **泳道顺序**：按流程逻辑从左到右排列角色泳道。
- **连线关系**：确保每个节点都有明确的进出连接，不要有线条重叠的情况。
- **跨泳道流转**：
- 当流程从一个角色传递到另一个角色时，连接线会自动跨越泳道。
- 不需要特殊标记，PlantUML会自动处理。
- **并行处理**：如遇并行任务，使用 `fork` 和 `fork again` 语法。
- **循环处理**：驳回/重试等情况，使用 `detach` 或明确指向。

##### 4. 内容简化要求
- **禁止异常分支**：忽略所有失败、重试、超时等异常处理。
- **禁止定时任务**：不包含等待X时间等节点。
- **禁止跨流程汇总**：不包含财务分析、报表生成等非本流程核心职责的节点。
- **避免重复**：确保每个节点在单一路径中不重复。

##### 5. 空值规则
若分析领域内不存在某类流程（如无子流程、CRUD流程或简单流程），则在输出中不生成该类型的流程条目。

---

#### 输出格式
- 必须是完整的PlantUML泳道图代码。
- 代码块以 `@startuml` 开始，`@enduml` 结束。
- 使用 `title` 指令设置图表标题。
- 使用泳道语法 `|角色名|`。

---

#### 示例（符合上述原则的正确写法）

```plantuml @startuml title 用车申请流程

|用车人|start:提交申请;

|派车管理员| // 合并了审批与派车职责:审核申请;if (是否通过？) then (是):安排车辆;:通知司机;

|司机|:确认出车;:完成行程并反馈;

|派车管理员|:确认行程结束;stopelse (否):驳回申请;

|用车人|:收到驳回;stopendif

@enduml
```

请严格按照以上全部原则和规范，对所提供的业务需求及角色列表进行分析，并输出所有识别出的业务流程PlantUML代码。
