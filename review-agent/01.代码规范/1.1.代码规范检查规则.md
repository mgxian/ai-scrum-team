## 代码规范检查规则

核心指令：根据以下代码变更内容，按照团队代码规范进行审查，输出审查报告。

代码变更如下：

【$code_diff】

项目技术栈如下：

【$tech_stack】

---

### 一、命名规范检查

| 检查项 | 规则 | 严重程度 |
|:--|:--|:--|
| 类名 | PascalCase，名词或名词短语 | 高 |
| 方法名 | camelCase，动词或动词短语 | 高 |
| 变量名 | camelCase，有意义的描述性名称，禁止单字母（循环变量除外） | 中 |
| 常量名 | UPPER_SNAKE_CASE | 中 |
| 文件名 | 与主导出类/组件名一致 | 低 |
| 数据库字段 | snake_case，与领域模型术语一致 | 高 |

### 二、代码结构检查

| 检查项 | 规则 | 严重程度 |
|:--|:--|:--|
| 方法长度 | 单个方法不超过 50 行（不含空行和注释） | 中 |
| 类长度 | 单个类不超过 500 行 | 中 |
| 方法参数 | 不超过 5 个参数，超过则封装为对象 | 中 |
| 嵌套深度 | if/for/while 嵌套不超过 3 层 | 高 |
| 圈复杂度 | 单个方法圈复杂度不超过 10 | 高 |
| 重复代码 | 相似代码块出现 2 次以上需抽取公共方法 | 中 |

### 三、注释与文档检查

| 检查项 | 规则 | 严重程度 |
|:--|:--|:--|
| 公共 API | 必须有文档注释（JSDoc / JavaDoc / docstring） | 高 |
| 复杂逻辑 | 非显而易见的业务逻辑必须有行内注释 | 中 |
| TODO/FIXME | 必须关联 Issue 编号或负责人 | 低 |
| 废弃代码 | 注释掉的代码必须删除，通过版本控制追溯 | 中 |

### 四、错误处理检查

| 检查项 | 规则 | 严重程度 |
|:--|:--|:--|
| 异常捕获 | 禁止空 catch 块，必须记录日志或重新抛出 | 高 |
| 异常粒度 | 禁止捕获顶层 Exception/Error，需捕获具体异常类型 | 高 |
| 返回值 | API 层必须返回统一错误结构，禁止返回裸异常堆栈 | 高 |
| 资源释放 | 数据库连接、文件句柄、流必须在 finally/using/try-with-resources 中释放 | 高 |
