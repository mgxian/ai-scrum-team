## 改进项提取与分发

核心指令：根据 Sprint 回顾报告，提取可落地的改进项，分类后生成知识条目草稿，供 PO 确认后写入各角色的规则/知识库。

Sprint 回顾报告如下：

【$retrospective_report】

各角色当前规则版本如下：

【$current_rules_version】

---

### 第一步：改进项提取

从回顾报告中识别所有可转化为规则或知识的改进项：

| 编号 | 来源问题 | 问题根因 | 改进项描述 | 出现频次 | 是否跨 Sprint 重复 |
|:--|:--|:--|:--|:--|:--|

筛选标准：
1. 同一问题在 2 个及以上 Sprint 中重复出现 → 必须固化为规则
2. 单次出现但影响程度为"高" → 建议固化
3. 单次出现且影响程度为"低" → 仅记录观察，不急于固化

### 第二步：改进项分类

将提取的改进项按目标角色分类：

| 编号 | 改进项 | 目标角色 | 知识类型 | 写入位置 |
|:--|:--|:--|:--|:--|

知识类型说明：
- **规则新增**：新增一条检查规则或约束条件
- **规则调整**：修改现有规则的阈值、权重或优先级
- **案例补充**：向 RAG 知识库添加典型案例（问题 + 解决方案）
- **流程变更**：调整工作流步骤或触发条件
- **模型校准**：调整估算参数或预警阈值

### 第三步：知识条目草稿

对每个改进项，生成结构化的知识条目：

```
条目编号：KF-{sprint_number}-{seq}
来源 Sprint：Sprint {N}
目标角色：{role_name}
知识类型：{type}
写入位置：{file_path}

当前状态：
  {描述当前规则或缺失情况}

改进内容：
  {具体要新增或修改的规则/知识内容}

生效条件：
  {什么情况下触发这条规则}

验证方式：
  {如何在下个 Sprint 验证这条改进是否生效}

状态：待 PO 确认
```

### 第四步：PO 确认

将所有知识条目草稿提交 PO 审批：

| 条目编号 | 改进内容摘要 | PO 决策 | 备注 |
|:--|:--|:--|:--|

PO 决策选项：
- **批准**：直接写入
- **修改后批准**：PO 提出修改意见，调整后写入
- **暂缓**：再观察一个 Sprint
- **拒绝**：属于偶发问题，不固化

### 第五步：写入与生效

批准的条目按以下规则写入：

| 目标角色 | 写入方式 | 生效时机 |
|:--|:--|:--|
| BA Copilot | 追加到对应步骤的 prompt 模板补充规则段 + RAG 知识库索引 | 下个 Refinement |
| Arch Copilot | 追加到对应模块的检查清单或 ADR 库 | 下个 Planning |
| Code Agent | 更新 Steering 规则或编码规范文档 | 立即生效 |
| Review Agent | 更新对应规则文件的检查项或权重 | 立即生效 |
| QA Agent | 更新回归测试必测清单或测试策略 | 下个 Execution |
| SM Agent | 更新估算系数或预警阈值 | 下个 Planning |
| Flow Agent | 更新流程编排规则 | 下个 Sprint |

### 第六步：生效验证

在下个 Sprint 的回顾中，检查已写入的改进项是否生效：

| 条目编号 | 改进内容 | 写入 Sprint | 验证 Sprint | 是否生效 | 效果说明 |
|:--|:--|:--|:--|:--|:--|

未生效的条目需分析原因，进入下一轮改进循环。
